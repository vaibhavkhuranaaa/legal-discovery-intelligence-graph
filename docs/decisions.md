# Architectural Decisions (ADR-lite)

Lightweight record of significant technical decisions. Each entry: context, decision,
alternatives considered, consequences. New significant choices must be recorded here.

---

## ADR-0001: Dependency and environment management via `uv`

**Context:** Need reproducible, isolated Python environments without touching system Python,
and a single source of truth for dependencies.

**Decision:** Use `uv` exclusively, driven by `pyproject.toml` + `uv.lock`. `pip` is prohibited.
`requirements.txt` exists only as a generated export (`uv export --format requirements-txt
--no-dev --no-hashes -o requirements.txt`) because Streamlit Community Cloud installs from it;
it is never hand-edited and never the source of truth.

**Alternatives considered:** `pip` + `venv` + hand-maintained `requirements.txt` (no lockfile,
drift risk — and drift between the lockfile and the deployed app would be invisible);
`poetry` (viable, slower, `uv` is the current standard).

**Consequences:** Contributors need `uv`. `requirements.txt` must be re-exported whenever
`uv.lock` changes, or deployment drifts from local — the deployment checklist includes this.

## ADR-0002: `src` layout with package name `legal_discovery_graph`, Hatchling backend

**Context:** Need an import layout that prevents accidentally importing from the working
directory instead of the installed package.

**Decision:** `src/legal_discovery_graph/` layout, Hatchling build backend, installed editable
via `uv sync`. Subsystems are subpackages (`ingestion`, `extraction`, `retrieval`, `graph`,
`evaluation`, `ui`).

**Consequences:** All tooling runs via `uv run` against the installed package; no `PYTHONPATH`
manipulation.

---

## ADR-0003: PostgreSQL + pgvector (Supabase) for semantic retrieval

**Context:** The system needs a cloud-hosted vector store for a publicly deployed app, plus
relational metadata (documents, chunks, mentions) for citations and evaluation.

**Decision:** One PostgreSQL database (Supabase-managed, free tier) holding both relational
metadata and embeddings via the pgvector extension, accessed with SQLAlchemy + psycopg.

**Alternatives considered:** Dedicated vector DBs (Pinecone/Weaviate/Qdrant Cloud) — a second
service and data store to keep consistent, while corpus scale (a few thousand chunks) is far
below where a specialized ANN service pays off; Chroma/FAISS local files — incompatible with a
stateless Community Cloud deployment and not demonstrably "production cloud" for portfolio
purposes.

**Consequences:** Citations, mention provenance, and vectors live in one transactional store —
one join away from each other. HNSW index and SQL filtering cover the scale; if the corpus ever
outgrows Postgres ANN, the retrieval module is the single seam to swap.

## ADR-0004: Neo4j AuraDB for the relationship graph (alongside, not instead of, pgvector)

**Context:** Investigation questions are relationship-shaped (who↔whom↔what↔when); variable-
depth traversals and subgraph rendering are awkward in SQL. This is also a Graph RAG portfolio
project — the graph leg must be first-class.

**Decision:** Model entities/documents/events as a property graph in Neo4j AuraDB Free, queried
with Cypher via the official Python driver. pgvector remains the retrieval store; Neo4j stores
relationships only, joined by shared IDs (see `DATA_MODEL.md`). Graph edges carry `chunk_id`
provenance.

**Alternatives considered:** Recursive CTEs / edge tables in Postgres (feasible, but multi-hop
expansion and graph visualization become the hardest code in the project instead of a Cypher
one-liner); embedded graph libs like NetworkX (in-memory, not a persistent cloud service,
undermines the deployed-architecture goal).

**Consequences:** Two stores must stay consistent — the shared-ID discipline and single loading
script are the mitigation. AuraDB Free instances pause when idle; the app must handle
reconnection gracefully (a defined failure boundary in `architecture.md`).

## ADR-0005: Fully synthetic, generator-produced corpus with emitted gold labels

**Context:** Real discovery material is confidential by nature; even public court exhibits
carry real names and cannot be freely redistributed or labeled cheaply. The project also needs
exact ground truth for precision/recall/F1.

**Decision:** The corpus is generated by seedable code (`scripts/bootstrap_data.py`) around a
coherent fictional investigation scenario. The generator emits gold labels (entities, events,
query→relevant-chunk sets) at creation time, since it knows what it planted. Generated data is
gitignored; the generator + seed are committed, making everything reproducible.

**Alternatives considered:** Enron email corpus (public, but real people's emails — ethically
awkward for a demo, and gold entity/retrieval labels would require expensive manual annotation);
LLM-generated corpus at runtime (non-deterministic, breaks reproducible evaluation).

**Consequences:** Zero data-privacy risk, perfect labels, deterministic metrics. Tradeoff:
synthetic text is cleaner than real discovery documents, so extraction metrics will look better
than they would in the wild — this is disclosed in `DATA_AND_EVALUATION.md` and the README.

## ADR-0006: Streamlit Community Cloud deployment; Streamlit + Plotly UI

**Context:** The project must be publicly deployed and demonstrable at zero/near-zero cost, and
the author's effort should concentrate on the Graph RAG engineering, not web infrastructure.

**Decision:** Streamlit for the dashboard, Plotly for the entity-graph and timeline
visualizations, deployed on Streamlit Community Cloud directly from GitHub. Secrets via
Streamlit secrets; managed data backends (ADR-0003/0004) because Community Cloud has no
persistent disk.

**Alternatives considered:** FastAPI + React on a PaaS (most flexible, but weeks of UI work
orthogonal to the skills being demonstrated); Hugging Face Spaces (viable fallback; Community
Cloud has first-class Streamlit support and GitHub-native deploys); Docker on a VPS (monthly
cost and ops burden for no portfolio gain).

**Consequences:** Free public URL, deploy-on-push. Constraints accepted: cold starts (embedding
model load must be cached and surfaced gracefully), 1 GB memory ceiling (keeps the embedding
model small — see `EMBEDDING_MODEL_NAME` default), and read-only app against pre-indexed
backends.

## ADR-0007: Hugging Face sentence-transformers for embeddings (no paid embedding API)

**Context:** Chunks and queries need embeddings both at indexing time (dev machine) and query
time (deployed app). A paid API (OpenAI et al.) adds a billed dependency and a secret to a free
public demo; the corpus is small.

**Decision:** Use a local sentence-transformers model (default
`sentence-transformers/all-MiniLM-L6-v2`, 384-dim) for all embedding, configured via
`EMBEDDING_MODEL_NAME`. The pgvector column dimension is tied to the model; changing models
requires a re-index and an ADR update.

**Alternatives considered:** OpenAI `text-embedding-3-small` (higher quality, but introduces
cost, rate limits, and an API key into a public demo); larger local models (better recall,
violates the Community Cloud memory ceiling per ADR-0006).

**Consequences:** Zero embedding cost, fully reproducible vectors, one fewer secret. Tradeoff:
retrieval quality below state-of-the-art APIs — acceptable and measured honestly by the
evaluation harness.

## ADR-0008: Deterministic uuid5 IDs and composer-tracked gold labels for generated data

**Context:** Milestone 1's exit criterion is byte-identical regeneration for a given seed, and
the evaluation methodology depends on exact gold labels. `models.py` defaults to random uuid4
IDs, and hand-annotating mention offsets after generation would be error-prone and expensive.

**Decision:** (a) Generated objects mint IDs via `uuid5` over stable key parts
(`legal_discovery_graph/ids.py`) — e.g. `("doc", seed, index)`, `("chunk", document_id,
sequence)` — while uuid4 remains the runtime default for non-generated objects. (b) Document
text is assembled through a `Composer` that records the exact character span of every entity
mention as the text is built (`datagen/composer.py`); scenario templates never write an entity
name as plain text, which is the discipline that makes recall labels complete. (c) Timestamps
come from the scenario clock, never `now()`. (d) The bootstrap writes raw files and then runs
the real ingestion pipeline over them (not an in-memory shortcut), so label resolution — mention
→ covering chunk, evidence snippet → containing chunk — is performed against exactly the chunks
any consumer will see, and serialization bugs surface immediately.

**Alternatives considered:** Post-hoc annotation by string search over generated text (breaks on
repeated surfaces and alias forms; offsets not guaranteed unique); seeding Python's `uuid4` (not
possible — it reads OS entropy); hashing content for IDs (IDs would change whenever wording is
tweaked, breaking label diffs across generator versions for unchanged documents).

**Consequences:** Same seed → byte-identical `data/raw/`, `data/processed/`, and `data/labels/`
(regression-tested). Chunks carry `start_char`/`end_char` document offsets in metadata, which
later milestones reuse for citation highlighting. Generator authors must follow the composer
mention discipline — a plain-text entity name in a template silently becomes a false negative in
the gold labels.
